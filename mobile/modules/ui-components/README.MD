| Level     | Description                          | Examples                           |
| --------- | ------------------------------------ | ---------------------------------- |
| Atoms     | Smallest building blocks             | `Text`, `Button`, `Icon`, `Input`  |

## ‚úÖ Component Guidelines

- All components must be **typed with TypeScript**
- Keep `atoms` and `molecules` **pure and stateless**
- Use `props` for customization ‚Äî no hardcoded styles or text
- Business logic lives outside components ‚Äî use redux-toolkit, hooks or containers

### üß† Redux Connection Guidelines for Expo Router + Atomic Design

| **Layer / Folder**       | **Connect to Redux?** | **Reason**                                                               |
| ------------------------ | --------------------- | ------------------------------------------------------------------------ |
| `components/atoms/`      | ‚ùå Never              | Pure UI elements (e.g., `Button`, `Text`). Should be fully reusable.     |
| `components/molecules/`  | ‚ùå Rarely             | Composed UI blocks ‚Äî should remain stateless and layout-focused.         |
| `components/organisms/`  | ‚úÖ Sometimes          | May handle data if reused across routes. Use hooks or props to inject.   |
| `components/templates/`  | ‚úÖ Optional           | Ideal for authenticated layouts, theme guards, or nested UIs.            |
| `app/` (screens)         | ‚úÖ Always             | Top-level routing layer in Expo Router ‚Äî safe to connect Redux directly. |
| `app/_layout.tsx`        | ‚úÖ Optional           | Good place for route guards, shared UI, or auth state checks.            |
| `hooks/`                 | ‚úÖ Recommended        | Encapsulate Redux logic for cleaner components and consistent access.    |
